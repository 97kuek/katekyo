# 文書運用規約

## 目的

設計資料を実装の後追いメモにせず、変更の意図、境界、検証方法をレビューできる状態に保つ。

## 情報の置き場所

1つの事実には1つの正本を割り当てる。

- 利用者が行うこと: `usecases.md`
- システムが守ること: `requirements.md`
- コンポーネントの責務と依存: `architecture.md`
- 永続化構造とDB制約: `data-models.md`
- 呼び出し契約: `api-spec.md`
- 検証方法: `testing-strategy.md`
- 複数案から選んだ理由: `adr/`

他文書では要約を重複させず、IDまたは相対リンクで正本を参照する。

## 識別子

| 接頭辞 | 対象 | 例 |
| --- | --- | --- |
| `UC-` | ユースケース | `UC-04` 宿題を提出する |
| `FR-` | 機能要件 | `FR-HW-02` 提出を受け付ける |
| `BR-` | ビジネスルール | `BR-HW-01` 許可された状態遷移 |
| `NFR-` | 非機能要件 | `NFR-SEC-01` テナント分離 |
| `ADR-` | 設計判断 | `ADR-0001` 文書の正本と追跡性 |
| `T-` | テストシナリオ | `T-SEC-01` 越境アクセス拒否 |

既存の `UC-01` から `UC-15` は変更しない。要件IDは領域単位で採番し、番号の再利用を禁止する。

## 変更時の更新契機

| 変更 | 同じPRで確認する文書 |
| --- | --- |
| ユーザーフロー | `usecases.md`、ユースケース図、E2Eシナリオ |
| 業務ルール・状態 | `requirements.md`、状態遷移図、単体／統合テスト |
| Prisma schema | `data-models.md`、クラス図、migration、テナント境界テスト |
| Server Action | `api-spec.md`、シーケンス図、認可／入力検証テスト |
| Route Handler・外部連携 | `api-spec.md`、コンポーネント／配置図、契約テスト |
| 依存方式・基盤の選択 | `architecture.md`、該当ADR |

## レビュー基準

- 要件は実装方法ではなく観測可能な振る舞いとして記述されている
- 受入条件に正常系、拒否条件、権限境界が含まれている
- 図と文章で名称、方向、状態名が一致している
- `teacherId`、`studentId`、`ParentStudent` のいずれが認可境界か明記されている
- 外部通知の失敗が主処理へ与える影響と再試行方針が明記されている
- 変更した要件に対応するテスト、またはテストしない理由がある
- ファイルパス、Action名、モデル名が現在のコードと一致している

## UML作成規約

- PlantUMLのソースを正本とし、手編集したSVGを作らない
- 1図につき1つの問いに答える。巨大な万能図を作らない
- 図の冒頭に対象範囲、関連ID、最終確認対象をコメントで記載する
- 実装クラスを示す図と業務概念を示す図を混在させない
- 色だけで意味を伝えず、ラベルやステレオタイプを併用する

## 廃止手順

文書を削除またはアーカイブする前に、現行情報が正本へ移されたことを確認する。履歴価値がある文書は `docs/archived/` に移し、先頭へ「Historical」と代替文書へのリンクを記載する。
